# Feast Dev Port-Forward Script

Automates port-forwarding of Feast Feature Store registry services for local dashboard development.

Instead of manually running one `kubectl port-forward` per Feature Store and updating `.env.local` by hand, this script discovers all enabled Feature Stores from the cluster and sets everything up in one command.

---

## Requirements

| Tool | Install |
|------|---------|
| `kubectl` or `oc` | Must be logged in to the cluster |
| `jq` | `brew install jq` |
| `python3` | Pre-installed on macOS |

---

## Quick Start

```bash
# From the repo root
./packages/feature-store/scripts/feast-dev-portforward.sh
```

The script will:
1. Find all namespaces labelled `opendatahub.io/feast=true`
2. Find all FeatureStore CRDs labelled `feature-store-ui=enabled` in those namespaces
3. Port-forward each registry REST service to `localhost` starting at port `6570`
4. Writes the required `FEAST_*` port variables to `.env.local` — one variable per FeatureStore, named after its service (e.g. `FEAST_FEAST_<NAME>_REGISTRY_REST_PORT`)
5. Stay running and monitor — prints `[DOWN]` if a forward dies. Press **Ctrl+C** or run `stop` from another terminal to shut everything down

## Commands

### Start port-forwarding

```bash
./packages/feature-store/scripts/feast-dev-portforward.sh
```

Discovers FeatureStores, starts port-forwards, and updates `.env.local`. Stays running and monitors forwards — **Ctrl+C stops everything cleanly**.

```bash
# Start from a different base port (default is 6570)
./packages/feature-store/scripts/feast-dev-portforward.sh --base-port 8440

# Skip updating .env.local
./packages/feature-store/scripts/feast-dev-portforward.sh --no-env-update

# Preview what would happen without making any changes
./packages/feature-store/scripts/feast-dev-portforward.sh --dry-run
```

---

### Check status

```bash
./packages/feature-store/scripts/feast-dev-portforward.sh status
```

Shows all active Feast port-forwards and whether each local port is reachable:

```
  feast-<name>-registry-rest  (namespace: <namespace>)
    PID      : 12345
    Port     : localhost:6570
    Status   : reachable
```

Run this from a second terminal while the script is running in the first.

---

### Stop forwards

```bash
# Kill all active Feast port-forwards
./packages/feature-store/scripts/feast-dev-portforward.sh stop

# Kill a specific one by CRD name (partial match is fine)
./packages/feature-store/scripts/feast-dev-portforward.sh stop <crd-name>
```

You can also just press **Ctrl+C** in the terminal where the script is running.

---

### Show usage

```bash
./packages/feature-store/scripts/feast-dev-portforward.sh --help
```

## How env vars are named

The backend looks up each Feast service by deriving an env var from the service name:

```
CRD name:     <my-store>
Service name: feast-<my-store>-registry-rest
Env var:      FEAST_FEAST_<MY_STORE>_REGISTRY_REST_PORT=6570
```

The fallback `FEAST_REGISTRY_SERVICE_PORT` is also written and used when no per-service var matches. It is set to the base port (first forwarded service's port).

---

## What gets written to .env.local

The script manages a clearly marked block in `.env.local`. It is safe to run multiple times — the block is replaced on each run, never duplicated.

```
## BEGIN feast-dev-portforward (auto-generated — do not edit this block manually)
# Generated by scripts/feast-dev-portforward.sh on <date>
# Cluster: <cluster-url>
# User:    <username>
# FEAST_REGISTRY_SERVICE_PORT is the fallback used when no per-service port var is set.
# It is set to the base port (<base-port>), which is the port of the first forwarded service.
FEAST_REGISTRY_SERVICE_PORT=6570
FEAST_FEAST_<STORE_A>_REGISTRY_REST_PORT=6570
FEAST_FEAST_<STORE_B>_REGISTRY_REST_PORT=6571
FEAST_FEAST_<STORE_C>_REGISTRY_REST_PORT=6572
## END feast-dev-portforward
```

All other content in `.env.local` is left untouched. The file is gitignored so these values are never committed.

## CRD name collisions

If two FeatureStore CRDs in **different namespaces** share the same name, they map to the **same env var** and only one will be reachable in local dev. The script detects this and warns you:

```
┌─────────────────────────────────────────────────────────────┐
│  WARNING: CRD name collision detected                       │
└─────────────────────────────────────────────────────────────┘
  CRD name '<name>' exists in multiple namespaces.
  Both map to env var: FEAST_FEAST_<NAME>_REGISTRY_REST_PORT
  Only one namespace will work in local dev (last port wins).
```

To fix, rename one of the FeatureStore CRDs on the cluster so they have distinct names.

## Two-terminal workflow

**Terminal 1** — start and leave running:

```bash
./packages/feature-store/scripts/feast-dev-portforward.sh
```

**Terminal 2** — manage from here:

```bash
./packages/feature-store/scripts/feast-dev-portforward.sh status
./packages/feature-store/scripts/feast-dev-portforward.sh stop <crd-name>
./packages/feature-store/scripts/feast-dev-portforward.sh stop
```

When a forward dies or is killed, Terminal 1 prints a `[DOWN]` line within 5 seconds. A heartbeat `[OK] N/N port-forward(s) active` is printed every 60 seconds so you know the monitor is still watching.

---

## Labelling requirements

The script only picks up FeatureStores that have been explicitly labelled for the UI:

```bash
# Label a namespace so the script scans it
kubectl label namespace <ns> opendatahub.io/feast=true

# Label a FeatureStore CRD to include it
kubectl label featurestore <name> -n <ns> feature-store-ui=enabled
```
